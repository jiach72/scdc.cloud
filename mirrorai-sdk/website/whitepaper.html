1. 摘要（Executive Summary）

AI Agent 正从实验室走向生产环境，但业界缺乏系统化的评测框架和行为审计能力。开发者面对"黑盒"问题：当 Agent 出现异常行为时，无法追溯根因；当需要证明 Agent 可靠性时，没有可量化的指标支撑。

明镜提出一套三层架构方案——评测层（Evaluation）、证据层（Evidence）与监控层（Monitoring），以"行为证据优于功能测试"为核心理念，通过 Blackbox 录制引擎实现零侵入的行为捕获，结合 Ed25519 签名链确保证据不可篡改，为 AI Agent 提供从开发到生产的全生命周期可观测性保障。

2. 问题陈述

2.1 AI Agent 从实验到生产的鸿沟

当前 AI Agent 开发面临严峻的工程化挑战。根据 2025 年 Gartner 调研，仅有 12% 的 Agent 项目能够从 POC 阶段成功进入生产部署。核心原因包括：

  • 行为不可预测：Agent 具有自主决策能力，相同输入可能产生不同输出，传统测试方法难以覆盖。
  • 调试成本极高：Agent 调用链通常涉及多次 LLM 推理、工具调用、条件分支，单次请求可能包含 20-50 个中间步骤，人工审查不可行。
  • 缺乏回归基线：模型更新（prompt 调整、模型版本升级）后，无法快速判断 Agent 行为是否退化。

2.2 "黑盒"问题：出了事找不到原因


用户投诉："Agent 给出了错误的退款建议"
    │
    ├── Prompt 变了？─────────────── 不确定
    ├── 模型版本更新了？────────────  可能
    ├── 工具调用参数错了？───────────  无法确认
    ├── 上下文窗口溢出了？───────────  没记录
    └── 系统提示词被绕过了？─────────  未知

没有完整的调用链记录，排查生产事故如同"盲人摸象"。团队在日志碎片中拼凑真相，MTTR（平均修复时间）往往超过 48 小时。

2.3 缺乏行业标准的评测方法

软件工程有单元测试、集成测试、端到端测试的成熟体系。但 AI Agent 的评测仍处于"手工评估"阶段：

现状 问题
人工打分 主观性强，不可复现，成本高
简单的正确性检查 只看最终答案，忽略推理过程
竞赛榜单（Benchmarks） 静态数据集，与生产场景脱节

业界缺少一个兼顾过程评测与结果评测、兼顾功能正确性与行为安全性的统一框架。

3.2 核心理念：行为证据 > 功能测试

方案 定位 局限
LangSmith LangChain 生态的调试工具 强绑定 LangChain，缺乏独立评测框架，不支持行为证据链
Langfuse 开源 LLM 可观测性 聚焦 token 级别的 trace，缺乏 Agent 行为维度的评测能力
Sentry 通用错误监控 不理解 Agent 调用语义，无法追踪推理链和工具调用关系
AgentOps Agent 监控 偏向监控而非评测，缺乏密码学保证的审计证据

传统软件测试关注"输出是否正确"，但 Agent 的"正确性"是多维度的。明镜的核心理念是：与其问"Agent 答对了没有"，不如问"Agent 是怎么做出决策的"。

这带来三个根本性转变：

  1. 从结果评测到过程评测：记录 Agent 的完整推理链、工具调用序列、上下文窗口变化。
  2. 从人工检查到自动化评分：基于 25 项可量化指标的自动评分，消除主观偏差。
  3. 从事后分析到实时监控：通过证据链实现生产环境的持续可观测性。

4. 评测方法论

4.1 五个评测维度的学术依据

明镜定义了五个核心评测维度，每个维度对应 Agent 能力的一个关键方面：

维度 学术依据 核心问题
可靠性 (Reliability) Huang et al., 2024; Software Reliability Engineering Agent 是否稳定地完成预期任务？
安全性 (Safety) Anthropic Constitutional AI; OWASP LLM Top 10 Agent 是否会产生有害或越权行为？
效率 (Efficiency) Park et al., 2023 (Voyager); Token Economics Agent 是否以合理成本完成任务？
可解释性 (Explainability) Molnar, 2020 (Interpretable ML); XAI Agent 的决策过程是否可理解？
鲁棒性 (Robustness) Mazeika et al., 2024 (AdvBench); Adversarial ML Agent 在对抗性输入下是否可靠？

这五个维度的选取基于两个原则：（1）覆盖 Agent 生产部署的全生命周期风险；（2）每个维度都有可量化、可自动化的评测方法。

4.2 25 项指标的设计原则

每个维度下设 5 项具体指标，总计 25 项。设计遵循以下原则：

  • 可自动化：每项指标可通过程序化方式评分，不依赖人工判断。
  • 可量化：输出为 0-100 的数值分数，支持跨版本、跨团队比较。
  • 可归因：低分指标能关联到具体的 Agent 行为事件（证据层支持）。
  • 独立性：指标之间正交，避免重复计分。

示例（可靠性维度）：


reliability:
  - task_completion_rate:    # 任务完成率
      formula: "completed_tasks / total_tasks * 100"
      weight: 0.25
  - output_consistency:      # 输出一致性
      formula: "1 - stddev(outputs_across_runs) / mean"
      weight: 0.25
  - error_recovery_rate:     # 错误恢复率
      formula: "recovered_errors / total_errors * 100"
      weight: 0.20
  - context_adherence:       # 上下文遵从度
      formula: "on_topic_responses / total_responses * 100"
      weight: 0.15
  - tool_call_accuracy:      # 工具调用准确率
      formula: "valid_tool_calls / total_tool_calls * 100"
      weight: 0.15

4.3 动态评测 vs 静态检查的权衡

类型 适用场景 优势 局限
静态检查 Prompt 模板审查、工具定义验证、系统提示词合规 速度快（<1s），无 LLM 成本 无法捕获运行时行为
动态评测 端到端对话测试、工具调用链验证、对抗性测试 覆盖真实场景 成本高，耗时长

明镜采用混合策略：静态检查作为 CI/CD 的门禁（每次提交触发），动态评测作为发布前的深度验证（每日/每周运行）。

4.4 评分模型的数学基础

综合评分采用加权几何平均数（Weighted Geometric Mean），而非算术平均数：


Score = ∏(score_i ^ w_i)

其中：
  score_i ∈ [0, 1]  (归一化后的维度得分)
  w_i ∈ [0, 1]      (维度权重，Σw_i = 1)

选择几何平均数的原因：

  1. 惩罚短板效应：任何一项维度得分为 0，总分为 0。这符合安全敏感场景的需求——安全性为零的 Agent 不应因其他维度优秀而获得高分。
  2. 维度间敏感性：几何平均数对低分项更敏感，鼓励均衡发展而非偏科。
  3. 学术先例：HDI（人类发展指数）和 EFQM（欧洲质量管理模型）均采用几何平均数。

5.2 录制引擎设计

零侵入原则：Agent 应用无需修改任何业务代码，通过 SDK 的 monkey-patching 或 middleware 注入机制实现自动录制。


# 用户集成代码（仅需 3 行）
from mirrorai import Blackbox

bb = Blackbox(api_key="lb_live_xxx")
bb.attach(my_agent)  # 自动注入录制钩子

低延迟保证：

  • 事件写入采用无锁环形缓冲区（Ring Buffer），默认大小 4096 事件。
  • 序列化采用二进制格式（MessagePack），比 JSON 快 3-5 倍。
  • 异步写入：录制操作不阻塞 Agent 主线程，延迟 < 0.5ms（P99）。

事件类型：


{
  "event_id": "evt_a1b2c3d4",
  "timestamp": "2026-03-24T07:30:00.123Z",
  "type": "llm_call",

    "latency_ms": 2340
  },
  "parent_id": "evt_x1y2z3",

5.3 脱敏引擎

PII 识别算法：采用三层检测策略：

  1. 正则模式层：检测邮箱、手机号、身份证号、银行卡号等结构化 PII。
  2. NER 模型层：基于轻量级 NER 模型识别人名、地址、组织名等非结构化 PII。
  3. 上下文推断层：通过字段名（如 ssn、password）推断敏感字段。

误报控制：每层检测输出置信度分数，三层综合置信度 > 0.85 时才执行脱敏。误报率控制目标：< 0.1%。

PII 类型 策略 示例
邮箱 保留域名 u*@example.com
手机号 保留前3后4 138**1234
人名 替换为 [NAME_N] [NAME_1]

设计目标：确保每条录制事件不可篡改、可独立验证、可形成签名链。

签名流程：


Event_N ──serialize──▶ Canonical JSON ──hash──▶ SHA-256 ──sign──▶ Signature_N
                                                            ▲
                                                            │
                                                    Ed25519 Private Key
                                                    (本地存储，永不上传)

签名链结构：


class SignedEvent:
    event: dict           # 事件内容（脱敏后）
    sequence: int         # 序列号
    prev_hash: str        # 前一事件的哈希（链式结构）
    content_hash: str     # 当前事件内容的 SHA-256
    signature: bytes      # Ed25519 签名
    public_key_id: str    # 公钥标识（用于验证）

密钥管理：

  • 私钥生成后存储在本地安全存储（macOS Keychain / Linux Secret Service / Windows DPAPI）。
  • 私钥永不离开用户设备，明镜服务端只接收公钥。
  • 支持密钥轮换：每 90 天自动提醒，旧公钥保留用于历史记录验证。

5.5 报告生成

报告生成采用确定性序列化：相同内容的事件序列无论处理顺序如何，都生成相同哈希的报告。


def generate_report(events: list[SignedEvent]) -> Report:
    # 1. 按 sequence 排序（确定性）
    sorted_events = sorted(events, key=lambda e: e.sequence)
    
    # 2. 验证签名链完整性
    for i, event in enumerate(sorted_events):
        assert verify_signature(event)
        if i > 0:
            assert event.prev_hash == sorted_events[i-1].content_hash
    
    # 3. 计算报告哈希
    canonical = deterministic_serialize(sorted_events)
    report_hash = hashlib.sha256(canonical.encode()).hexdigest()
    
    return Report(events=sorted_events, hash=report_hash)

6.2 私钥永不上传的设计

这是明镜安全模型的核心假设：

  • 私钥生成：在用户设备上通过操作系统安全随机数生成器（/dev/urandom 或 CryptGenRandom）生成。
  • 私钥存储：优先使用平台安全存储（Keychain / Secret Service / DPAPI），降级方案为 AES-256-GCM 加密文件，密钥由用户主密码派生（PBKDF2, 100,000 迭代）。
  • 私钥使用：签名操作在本地完成，私钥内存中的明文在签名后立即清零（memset_s）。
  • 架构保证：即使明镜云端被完全入侵，攻击者也无法伪造或篡改历史事件。

10. 结论

AI Agent 的生产化部署正在加速，但缺乏系统化的评测和审计能力成为最大的工程化瓶颈。明镜以行为证据为核心，构建了从评测到监控的完整技术栈：

  1. 评测方法论提供了可量化、可自动化的 25 维度评估框架，解决了"Agent 好不好"的度量问题。
  2. Blackbox 技术以零侵入方式捕获 Agent 完整行为链，配合 Ed25519 签名确保证据不可篡改，解决了"Agent 干了什么"的追溯问题。
  3. 安全与隐私设计确保敏感数据不出设备，满足 PIPL/GDPR 合规要求，解决了"敢不敢用"的信任问题。

我们相信，可评测的 AI 才是可信赖的 AI。明镜致力于成为 AI Agent 时代的评测基础设施，让每一个 Agent 都有据可查、有证可验。
